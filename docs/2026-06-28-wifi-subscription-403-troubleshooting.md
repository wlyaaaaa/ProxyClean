## 0. TL;DR(结论先行)

**真正的根因:机场服务器对【你家宽带公网 IP】返回 `HTTP 403`、拒绝下发订阅;但对【代理落地节点 IP(东京)】和【手机蜂窝 IP】返回 `200`、正常下发。**

- 所以"更新节点"在 **WiFi 上失败**(尤其掉到"暂无代理"空状态、只能走直连时),在**手机热点上成功**。
- **铁证**:同一订阅 token、同一条 `curl`,唯一区别是源 IP——
  - 直连(家宽 IP 出口)→ `HTTP 403`,收到 153 字节(一个拒绝页);
  - 经代理(东京节点 IP 出口)→ `HTTP 200`,收到 12093 字节(真订阅内容)。
- **有效解(永远管用)**:手机开热点 → 在蜂窝下点"更新节点" → 切回 WiFi。缓存到的节点在 WiFi 上照常能用。
- **永久解**:重启路由器/光猫换一个家宽公网 IP;还不行就找机场客服把该 IP 放行。
- **关键认知**:本故障是**控制面(取订阅/节点列表)**坏了,不是**数据面(走代理上网)**坏了。数据面一直是好的(实测经节点下载 3MB / 647KB·s⁻¹ 毫无问题)。两者被长期混为一谈,是排查走弯路的根源。

## 2. 症状(用户观察)

1. 电脑/手机一开始连飞鸟节点都正常;**自从点了"更新节点"后变 timeout**,且之后**连更新都做不了**。
2. **蜂窝数据永远正常;唯独家庭 WiFi 出怪事。**
3. 手机即使代理在跑,系统仍提示"**当前 WiFi 无法上网**"。
4. **部分场景**坏:`boxuegu.com`(博学谷)、`gemini`、B站**网页**打不开;但**百度/必应**、B站 **App** 正常。

6. **唯一稳定的破解法**:手机热点更新节点 → 切 WiFi 即恢复。

## 4. 真正的根因(证据链)

复现飞鸟"更新节点"那一下(取订阅),用真实订阅地址,在**家庭 WiFi** 上:

```
订阅域名解析:系统(WiFi)= 198.18.0.46(fake-ip)   DoH真实 = 111.48.30.50
直连取订阅:  HTTP=403  字节=153    耗时≈0.6s     ← 家宽 IP 被机场拒绝
经7892取订阅: HTTP=200  字节=12093  耗时≈0.06s    ← 东京节点 IP 被机场接受
API主机 /user/getSubscribe 直连可达 = 200
```

**解读**:
- 机场服务器**按源 IP 区别对待**:家宽公网 IP → `403`;东京节点 IP / 蜂窝 IP → `200`。
- 这跟 DNS 模式(fake-ip / redir-host)**无关**(两者都这样),跟节点、路由、MTU、IPv6 **都无关**(全部已排除)。

**为什么"更新节点"在 WiFi 上必失败 —— 死锁**:
1. 点"更新节点"时,飞鸟会先清空当前配置 → UI 显示"**暂无代理**";
2. 此刻**没有节点 = 没有代理可走** → 取订阅只能**走直连** → 家宽 IP → `403`;
3. 失败 → 仍是"暂无代理"或回退本地 → **你永远无法从空状态在 WiFi 上把订阅拉回来**。

**为什么"手机热点"能破解**:蜂窝是**另一个 IP**,机场不拦它 → 直连取订阅 = `200` → 成功;拿到的列表缓存本地 → 切回 WiFi 后数据面照常工作。

**为什么"折腾前正常"**:该家宽 IP 很可能是**最近才被机场风控/WAF 拉黑**——极可能是排查期间**反复点更新失败太多次**触发了 IP 级限流/封禁。所以它是**后天获得**的封禁,不是运营商行为。

## 5. 解决方案

### 5.1 立即可用(workaround,空状态下的唯一解)
**手机开热点 → 在蜂窝下点"更新节点" → 成功后切回 WiFi。** 电脑可连手机热点做同样的事。
> 原理:用没被封的蜂窝 IP 把订阅"灌"回本地缓存;之后 WiFi 上用节点本身没问题。

### 5.2 永久修复(让 WiFi 也能更新)
1. **换家宽公网 IP**:重启小米路由(`192.168.31.1` → 重启),或**光猫一起断电 5–10 分钟**(移动宽带 IP 较粘,断久才换)。换到没被拉黑的新 IP → WiFi 直接能更新。
   - 验证:换完用 `curl` 重测那个"直连取订阅"是不是从 403 变 200(命令见 §7)。
2. **别再在 WiFi 上反复戳"更新"**——每次失败可能续上封禁。
3. **找机场客服**:报上当前公网 IP,"这个 IP 取订阅一直 403,帮我放行"。
4. **若飞鸟有"通过代理更新订阅/更新走代理"开关**:打开它**仅在你当前还有可用节点时**有效(经代理=200);空状态下无效(无代理可走)。

### 5.3 手机(无法远程操作,需用户在机上点)
手机是独立的飞鸟实例,**同一个病**:在蜂窝下更新一次拿干净配置;DNS 模式**保持 fake-ip**(别学电脑去改 redir-host,见 §6.2)。

## 6. 重要的次要发现

### 6.1 TAG 的配置文件全加密,无法手改
`com.tag.lab` 下 `tag.config.yaml` / `settings.yaml` / `runtime.secure.yaml` / `subscription.yaml` **全部是 `TAG_ENC_V2_*` 密文**。**手改任何一个都会损坏它** → TAG 只能在其**自带 GUI** 里设置(模式选规则、若有 DNS 模式同理)。
- TAG **本就独立**:自己的核心 `mihomo-tag.exe`、端口 `7890`、自己的 DNS,不依赖飞鸟。
- **唯一禁忌:飞鸟与 TAG 不要同时开 TUN**——两者都抢 `198.18.0.1/16` fake-ip + 默认路由,必冲突。**用谁开谁。**

### 6.2 fake-ip 才是对的,redir-host 会坑(重要)
- **fake-ip**:把域名原样带到**落地节点**再解析 → 海外域名在境外解析,**绕开国内 DNS 污染**。这是这类客户端默认 fake-ip 的根本原因。
- **redir-host**:在**国内本地**解析 → google/gemini 等被 GFW 污染 → `ERR_CONNECTION_CLOSED`。本次实测踩坑,已撤回。
- 国内冷门站(boxuegu)绕海外的问题,**用"在蜂窝拉一份干净的完整订阅/geosite"解决即可**,代价远小于换 redir-host。

### 6.3 "测试延迟全 Timeout" 往往是误报
飞鸟"测试延迟"会**同时**朝所有节点(:10346 高端口)发起一大批连接,容易被家宽防扫描策略瞬时拦掉 → 测速里全红;但**正常单节点持续上网是通的**(实测 3MB 下载)。**判断好坏别看测速,直接开浏览器/`curl` 实测数据面。**

### 6.4 "使用本地配置更新成功(网络请求失败)" 这句的真意
飞鸟把两半硬塞一句:**「网络请求失败」= 在线拉订阅没成功;「使用本地配置更新成功」= 回退用了本地缓存、加载成功。** 即:在线更新挂了,本地兜底成功,节点能用但不是最新。

### 6.5 ⚠️ 安全提醒
- 飞鸟把**账号邮箱与密码以明文**存于 `shared_preferences.json`,且密码偏弱、疑似与 QQ 复用。**建议换强密码、勿复用。**
- 订阅 `token` / JWT 也在该文件明文存放——分享配置或截图时务必打码。

## 9. 验证结果(2026-06-28 同日 —— 根因与解法双重确认)

重启小米路由 **+ 光猫一起断电再开**后,家宽公网 IP 从 `120.243.147.x` 变为 `120.243.146.x`(同段、仅末段不同 → 证明是**单 IP 被封,非整段**)。**未改任何客户端设置、未更新节点**,仅换 IP,复测:

| 测项 | 换 IP 前 | 换 IP 后 |
|---|---|---|
| 直连取订阅(家宽 IP) | `403` / 153B | **`200` / 12093B** |
| 经代理取订阅(节点 IP) | `200` / 12093B | `200` / 12093B |
| `boxuegu` 直连访问 | 曾 `000`(后随干净配置转 `200`) | `200` |

**结论坐实**:同一机场、同一 token、同一命令,**唯一变量是家宽 IP**,`403` 随旧 IP 一并消失。根因 = "机场封了那个特定家宽 IP",**换 IP 即解,此后 WiFi 可直接"更新节点",无需再切蜂窝**。

> ⚠️ 动态 IP。旧 IP 极可能是被"反复失败的更新请求"触发风控拉黑。**勿无脑狂点更新**,以免新 IP 也中招。
>
> 📌 结构补记:小米路由 WAN 口是私网 `192.168.1.x`(网关 `192.168.1.1`)→ **光猫在拨号、双层 NAT**,真正公网 IP 在**光猫**上;故换 IP 必须**连光猫一起断电**才有效。
