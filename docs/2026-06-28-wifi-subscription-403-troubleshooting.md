## 0. TL;DR(结论先行)

**真正的根因:机场服务器对【你家宽带公网 IP】返回 `HTTP 403`、拒绝下发订阅;但对【代理落地节点 IP(东京)】和【手机蜂窝 IP】返回 `200`、正常下发。**

- 所以"更新节点"在 **WiFi 上失败**(尤其掉到"暂无代理"空状态、只能走直连时),在**手机热点上成功**。
- **铁证**:同一订阅 token、同一条 `curl`,唯一区别是源 IP——
  - 直连(家宽 IP 出口)→ `HTTP 403`,收到 153 字节(一个拒绝页);
  - 经代理(东京节点 IP 出口)→ `HTTP 200`,收到 12093 字节(真订阅内容)。
- **兜底解(永远管用)**:手机开热点 → 在蜂窝下点"更新节点" → 切回 WiFi。缓存到的节点在 WiFi 上照常能用。
- **更优解(无需手机,见 §9.7)**:用支持"按链接导入订阅"的 clash 客户端导入 `:443` 的 `subscribe?token=` URL(实测新 IP=`200`),**绕开飞鸟出问题的 `:5413` 面板**。(§9.4"关 TUN 直连更新"已被 §9.5 推翻:失败点是 `:5413` 端口被 RST,不是清代理死锁;RST 来源未定死,见 §9.6。)
- **关于换 IP**:重启路由器**+光猫**只能让"机场对该 IP 的 `403`"消失(端点层面),**并不能让"更新节点"按钮在 WiFi 上自愈**(那是 app 的死锁,§9.3)。实在不行找机场客服放行 IP。
- **⚠️ 观测盲区(§9.2)**:Claude 经飞鸟才连得上服务器,只能采到"已好"状态、**无法证伪故障态**——一切以用户实测为准。
- **关键认知**:本故障是**控制面(取订阅/节点列表)**坏了,不是**数据面(走代理上网)**坏了。数据面一直是好的(实测经节点下载 3MB / 647KB·s⁻¹ 毫无问题)。两者被长期混为一谈,是排查走弯路的根源。

## 4. 真正的根因(证据链)

复现飞鸟"更新节点"那一下(取订阅),用真实订阅地址,在**家庭 WiFi** 上:

```
订阅域名解析:系统(WiFi)= 198.18.0.46(fake-ip)   DoH真实 = 111.48.30.50
直连取订阅:  HTTP=403  字节=153    耗时≈0.6s     ← 家宽 IP 被机场拒绝
经7892取订阅: HTTP=200  字节=12093  耗时≈0.06s    ← 东京节点 IP 被机场接受
API主机 /user/getSubscribe 直连可达 = 200
```

**解读**:
- 机场服务器**按源 IP 区别对待**:家宽公网 IP → `403`;东京节点 IP / 蜂窝 IP → `200`。
- 这跟 DNS 模式(fake-ip / redir-host)**无关**(两者都这样),跟节点、路由、MTU、IPv6 **都无关**(全部已排除)。

**为什么"更新节点"在 WiFi 上必失败 —— 死锁**:
1. 点"更新节点"时,飞鸟会先清空当前配置 → UI 显示"**暂无代理**";
2. 此刻**没有节点 = 没有代理可走** → 取订阅只能**走直连** → 家宽 IP → `403`;
3. 失败 → 仍是"暂无代理"或回退本地 → **你永远无法从空状态在 WiFi 上把订阅拉回来**。

**为什么"手机热点"能破解**:蜂窝是**另一个 IP**,机场不拦它 → 直连取订阅 = `200` → 成功;拿到的列表缓存本地 → 切回 WiFi 后数据面照常工作。

**为什么"折腾前正常"**:该家宽 IP 很可能是**最近才被机场风控/WAF 拉黑**——极可能是排查期间**反复点更新失败太多次**触发了 IP 级限流/封禁。所以它是**后天获得**的封禁,不是运营商行为。

## 5. 解决方案

### 5.1 立即可用(workaround,空状态下的唯一解)
**手机开热点 → 在蜂窝下点"更新节点" → 成功后切回 WiFi。** 电脑可连手机热点做同样的事。
> 原理:用没被封的蜂窝 IP 把订阅"灌"回本地缓存;之后 WiFi 上用节点本身没问题。

### 5.2 永久修复(让 WiFi 也能更新)
1. **换家宽公网 IP**:重启小米路由(`192.168.31.1` → 重启),或**光猫一起断电 5–10 分钟**(移动宽带 IP 较粘,断久才换)。换到没被拉黑的新 IP → WiFi 直接能更新。
   - 验证:换完用 `curl` 重测那个"直连取订阅"是不是从 403 变 200(命令见 §7)。
2. **别再在 WiFi 上反复戳"更新"**——每次失败可能续上封禁。
3. **找机场客服**:报上当前公网 IP,"这个 IP 取订阅一直 403,帮我放行"。
4. **若飞鸟有"通过代理更新订阅/更新走代理"开关**:打开它**仅在你当前还有可用节点时**有效(经代理=200);空状态下无效(无代理可走)。

### 5.3 手机(无法远程操作,需用户在机上点)
手机是独立的飞鸟实例,**同一个病**:在蜂窝下更新一次拿干净配置;DNS 模式**保持 fake-ip**(别学电脑去改 redir-host,见 §6.2)。

## 6. 重要的次要发现

### 6.1 TAG 的配置文件全加密,无法手改
`com.tag.lab` 下 `tag.config.yaml` / `settings.yaml` / `runtime.secure.yaml` / `subscription.yaml` **全部是 `TAG_ENC_V2_*` 密文**。**手改任何一个都会损坏它** → TAG 只能在其**自带 GUI** 里设置(模式选规则、若有 DNS 模式同理)。
- TAG **本就独立**:自己的核心 `mihomo-tag.exe`、端口 `7890`、自己的 DNS,不依赖飞鸟。
- **唯一禁忌:飞鸟与 TAG 不要同时开 TUN**——两者都抢 `198.18.0.1/16` fake-ip + 默认路由,必冲突。**用谁开谁。**

### 6.2 fake-ip 才是对的,redir-host 会坑(重要)
- **fake-ip**:把域名原样带到**落地节点**再解析 → 海外域名在境外解析,**绕开国内 DNS 污染**。这是这类客户端默认 fake-ip 的根本原因。
- **redir-host**:在**国内本地**解析 → google/gemini 等被 GFW 污染 → `ERR_CONNECTION_CLOSED`。本次实测踩坑,已撤回。
- 国内冷门站(boxuegu)绕海外的问题,**用"在蜂窝拉一份干净的完整订阅/geosite"解决即可**,代价远小于换 redir-host。

### 6.3 "测试延迟全 Timeout" 往往是误报
飞鸟"测试延迟"会**同时**朝所有节点(:10346 高端口)发起一大批连接,容易被家宽防扫描策略瞬时拦掉 → 测速里全红;但**正常单节点持续上网是通的**(实测 3MB 下载)。**判断好坏别看测速,直接开浏览器/`curl` 实测数据面。**

### 6.4 "使用本地配置更新成功(网络请求失败)" 这句的真意
飞鸟把两半硬塞一句:**「网络请求失败」= 在线拉订阅没成功;「使用本地配置更新成功」= 回退用了本地缓存、加载成功。** 即:在线更新挂了,本地兜底成功,节点能用但不是最新。

### 6.5 ⚠️ 安全提醒
- 飞鸟把**账号邮箱与密码以明文**存于 `shared_preferences.json`,且密码偏弱、疑似与 QQ 复用。**建议换强密码、勿复用。**
- 订阅 `token` / JWT 也在该文件明文存放——分享配置或截图时务必打码。

## 9. 后续验证、观测盲区与最终解(2026-06-28 同日,深度复盘)

### 9.1 换 IP 只解了"端点层面",没解"更新按钮"
重启小米路由 **+ 光猫一起断电再开**后,家宽公网 IP 从 `120.243.147.x` → `120.243.146.x`(同段、仅末段不同 → **单 IP 被封,非整段**)。复测端点:

| 测项 | 换 IP 前 | 换 IP 后 |
|---|---|---|
| **直连**取订阅(家宽 IP) | `403`/153B | **`200`/12093B** |
| 经代理取订阅(节点 IP) | `200`/12093B | `200`/12093B |
| `boxuegu` 直连 | 曾 `000` | `200` |

→ "机场封了那个特定家宽 IP"**坐实**,换 IP 后**直连取订阅恢复 200**。
**但是**:飞鸟界面的 **"更新节点"按钮在 WiFi 上依旧失败**(`网络请求失败 / 未知网络错误`)。
**所以"换 IP=WiFi 能直接更新"是过度断言,已被实测推翻、特此更正。**

### 9.2 ⚠️ 观测盲区(本次最重要的方法论教训)
**Claude(本次排查工具)本身是经飞鸟代理才连得上服务器的。** 推论:
- 用户能与 Claude 对话 ⟺ 飞鸟此刻**必有可用节点**;
- 飞鸟真坏(冷启动/无节点)时,用户**连不上 Claude**,故障态对 Claude **结构性不可见**;
- 于是 Claude 每次诊断采到的**必然是"已好"的状态**(且是用户刚用手机热点 bootstrap 之后)。
→ **所有"200/通过"都带幸存者偏差;远程诊断不能用来"验证修好没",一切以用户实测为准。**

### 9.3 死锁模型(为什么"更新节点"按钮在 WiFi 上必挂)
1. 飞鸟取订阅默认**走自己的代理**(抗封);
2. 点"更新节点" → 核心重载 → **代理被清空("暂无代理")**、内置 DNS(`0.0.0.0:1053`)/TUN 同时短暂塌掉;
3. 取订阅请求正好掉进这个**真空窗口**(既无代理、又遇 DNS/路由空档)→ `未知网络错误`;
4. 失败 → 一直"暂无代理" → 用户被踢下线 → 只能手机 bootstrap。
> `curl` 直连能 `200`,是因为它在核心**稳定**时跑(a047 走 CN 直连规则即达);app 的失败发生在**重载真空窗口** —— 两者不矛盾。

### 9.4 ✅ 最终解(无需手机,把"别走代理"思路操作化)
**实证**:直连(不经代理)取整份订阅 = `200` / **12093 字节有效配置**。于是让飞鸟**别在"清代理"的窗口里取**:

> **关掉飞鸟 TUN/代理(或切"直连"模式)→ 点"更新节点"→ 成功后再开回 TUN。**

- 关 TUN = 没有代理可清、没有真空窗口,取订阅走**纯直连** → 新家宽 IP = `200` → 节点灌入;再开回 TUN 正常用,**全程不碰手机**。

**诚实声明**:受 §9.2 观测盲区限制,Claude **无法端到端验证**此法(真去试 Claude 只会看到"好");但直连路径**已实证可达**、且**失败也安全**(尚有本地缓存节点、可退回手机 bootstrap),值得用户亲手一试。**若此法仍不灵,手机热点 bootstrap 是兜底唯一解。**

### 9.5 🎯 最深一层根因:面板 API 端口 `5413` 被家宽拒绝(§9.4 实测后推翻 §9.4)
用户按 §9.4 关掉 TUN 后点"更新节点",拿到**决定性报错**:
```
SocketException: 远程计算机拒绝网络连接 (errno=1225 / WSAECONNREFUSED)
address = fbesa.apiv2.a047.com, port = 5413
```
- **飞鸟"获取节点信息"连的是机场面板 API 的 `5413` 端口(非标高端口),不是 443。** 之前全部测在 `:443`,**测错了地方**。
- 关 TUN 后 `:5413` 走**纯直连** → **被家宽 RST 主动拒绝**(不是超时)。所以 §9.4"关 TUN 直连更新"**也不成立**,特此推翻。
- 这彻底解释了为什么客户端怎么调都没用:
  - 关 TUN → `:5413` 直连被家宽拒;
  - 开 TUN → 更新清代理(§9.3 死锁)无出口;
  - 经代理也带不动 `:5413`(`curl -x 7892` = `000`);
  - 蜂窝放行 `:5413` → **只有手机热点这条路成立**。
- ⚠️ 又一次观测盲区:Claude 的 `Test-NetConnection :5413` 显示 `OPEN` 是 **TUN 假象**(TUN 本地把 TCP 接住了),**以 app 关 TUN 后的"拒绝"为准**。

### 9.6 ⚖️ 修正:别急着下"客户端无解/换宽带"(RST 来源未定死)
**§9.5 那句"家宽封 5413、客户端无解"下得太死,修正。** 硬漏洞:**家宽公网 IP(`120.243.x`)、面板服务器(`111.48.x`)、手机蜂窝——三者都是中国移动**;同一运营商宽带被 RST、蜂窝却通,"运营商封端口"讲不通。RST 真实来源候选(部分用户可修):

| 候选来源 | 用户可修? |
|---|---|
| (a) **光猫防火墙**(双层 NAT、光猫在拨号,移动光猫常拦非标出站口) | 可:光猫超管放行 / **改桥接让路由拨号**(去掉这层) |
| (b) **机场对"住宅宽带 IP"在 `:5413` 做 RST**(反滥用,逼用手机端) | 机场侧不可改,但可用 §9.7 绕过 |
| (c) **本地飞鸟残留**(关 TUN 时核心/系统代理 `127.0.0.1:7892`、fake-ip 没干净退出,RST 来自本地) | 可:彻底结束 `FlyingBirdCore` + `flushdns` 再试 |

**定位实验(去掉一切飞鸟/TUN 干扰)**:任务管理器结束 `FlyingBirdCore` → `ipconfig /flushdns` → 浏览器/curl 访问 `https://fbesa.apiv2.a047.com:5413/` → 仍 **refused**=真在网络/机场层;能**连上**(哪怕报错页)=**本地飞鸟**所致(可修)。

### 9.7 ✅ 优先试:绕开 `:5413` 面板,直接导 `:443` 订阅
**节点列表本体在 `:443` 的 `subscribe?token=` URL 上,实测新 IP = `200` / 12093 字节有效配置。** 飞鸟非要走 `:5413` 面板登录才中招;**换一个支持"按链接导入订阅"的 clash 客户端**(Clash Verge / mihomo Party / Clash for Windows 等),导入这条 `:443` URL → **根本不碰 `:5413`** → 家宽直接能更新。成本最低、最可能一劳永逸。

### 9.8 关于"换新宽带"
**不必,且是最后下策。** 优先级:① 手机热点 bootstrap(立即可用)→ ② §9.7 导 `:443` 订阅到别的 clash 客户端 → ③ §9.6(c) 干净重启飞鸟 + flushdns → ④ §9.6(a) 光猫改桥接/放行 5413 → 最后才换宽带(若 RST 是机场或光猫所致,换宽带未必解)。
